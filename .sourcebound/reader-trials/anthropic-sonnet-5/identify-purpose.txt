1. Who sourcebound applies to: maintainers of a repository whose code
   changes faster than its documentation -- i.e. the people responsible
   for keeping docs and source in sync as the source evolves.

2. The problem it addresses: stale claims in documentation -- prose that
   no longer matches the source it describes, with no mechanical way for
   a maintainer to tell which claims have gone stale as the code moves
   on.

3. The capability it gives that reader: a local, deterministic path from
   a source change to repaired, verified documentation. A model may be
   used to phrase the resulting facts in prose, but it never chooses
   what the facts are and never gates the pass/fail outcome -- that
   determination is deterministic and local.
