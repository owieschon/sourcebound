Answer: No. sourcebound itself does not revoke host network access for an
allowlisted (declared) process. Its process controls -- a disposable
copy of the repository, a temporary home and temp directory, a
restricted PATH, NO_COLOR, a timeout, a 1 MB combined I/O limit, symlink
rejection, and secret-output rejection -- constrain what a declared
command or plugin can read/write and how long it can run, but they are
explicitly not an operating-system sandbox. A declared executable can
still open a network connection (or reach an absolute host path) if the
surrounding runner/host permits it. Network isolation is therefore the
responsibility of the execution environment, not sourcebound: operators
who want to actually block network access for untrusted declared code
must run it inside a network-blocked container or equivalent OS sandbox
around sourcebound.

exact citation: docs/SECURITY_MODEL.md, "## Host boundary" section
(lines 24-26):
"The process controls are not an operating-system sandbox. A declared
executable can open a network connection or address an absolute host
path if the surrounding runner permits it. Run untrusted declared code
in a network-blocked container or equivalent OS sandbox."
