ANSWER:
1. Who sourcebound applies to:
   Maintainers of a repository whose code changes faster than its
   documentation -- i.e. maintainers facing code-to-documentation drift.

2. The problem it addresses:
   Because the code outruns the docs, documentation claims go stale, and the
   reader has no mechanical way to tell which claims are now false. The first
   body block frames sourcebound as the thing that "identifies stale claims,"
   which is exactly the gap it fills: without it, the maintainer/reviewer
   cannot reliably identify which documented claims have become stale.

3. The capability it gives that reader:
   A local, deterministic path from a source change to repaired, verified
   documentation -- it identifies the stale claims and drives them back to a
   verified state. The facts are owned by the source (and the checks), not by
   a model: models may phrase facts but never choose them or gate the result.

This maps to the rubric: maintainers facing code-to-documentation drift, the
inability to tell which claims are stale, and a local deterministic path to
repair and verify documentation.
