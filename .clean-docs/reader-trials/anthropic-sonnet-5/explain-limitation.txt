Answer: No. clean-docs itself does not revoke host network access
globally, and it does not revoke it even for the declared/allowlisted
processes it runs. Its process controls -- a disposable copy of the
repository, a temporary home and temp directory, a restricted PATH,
NO_COLOR, a timeout, a 1 MB combined I/O limit, symlink rejection, and
secret-output rejection -- constrain what a declared command or plugin
can read/write and how long it can run, but they are explicitly not an
operating-system sandbox. A declared executable can still open a
network connection (or reach an absolute host path) if the surrounding
runner/host permits it. Network isolation is therefore the
responsibility of the execution environment, not clean-docs: operators
who want to actually block network access must run untrusted declared
code inside a network-blocked container or equivalent OS sandbox around
clean-docs.

exact citation: docs/SECURITY_MODEL.md, "## Host boundary" section
(lines 24-26):
"The process controls are not an operating-system sandbox. A declared
executable can open a network connection or address an absolute host
path if the surrounding runner permits it. Run untrusted declared code
in a network-blocked container or equivalent OS sandbox."
